================================================================================
How the Controller layer collaborates with other modules (course project MVC skeleton documentation)
================================================================================

I. Roles
  - controller package: fx:controller for JavaFX FXML, responsible for "what the user clicked -> which business API to call".
  - Do not write Monopoly-specific settlement rules; rules belong in logic / model.
  - Complex "how to draw" is split into the view package (custom Node, GameplayUiBundle, GameplayViewCoordinator).

II. Relationship with FXML / Stage
  - Each *Screen.fxml points to a *Controller class in this package via fx:controller.
  - After ScreenNavigation.show(...) loads FXML, if the controller implements StageAware, the JavaFX Stage is injected,
    making it easy to switch Scenes within the same window.
  - The same StageAware injection is done in MonopolyDealApplication.start for the first screen.

III. Relationship with logic (GameEngine / GameLogic)
  - Entry points like lobby, bots, load save should call the corresponding method on AppContext.get().gameEngine()
    (e.g., startLocalGame, startLanGame) when "start game" is triggered, then navigate to GameplayScreen.
  - In-game buttons (play card, end turn, etc.) should submit intents through gameEngine().getGameLogic()
    or a later-encapsulated Command API; do not directly modify GameSession internal collections in the controller
    unless logic provides an API.

IV. Relationship with model
  - The controller only reads GameSession / DTO returned through logic; it is not treated as the single source of truth.
  - After session update: logic callback or GameObserver notifies viewCoordinator.refreshFromSession(...).

V. Relationship with view
  - AbstractGameplayScreenController constructs GameplayViewCoordinator in initialize,
    passing GameplayUiBundle (various Pane references) to a pure Java view coordinator class,
    later connecting to GameObserver / bindings.

VI. Relationship with persistence
  - Load save: LoadGameScreenController uses AppContext.get().saveGameService().load(Path),
    then passes the resulting GameSession to GameEngine for restoration (waiting on logic implementation).
  - Save: Call saveGameService().save(...) in the game menu or shortcut.

VII. Relationship with network (online)
  - In AddPlayerScreenController, room/join logic should ultimately connect to NetworkClient / GameServer;
    the UI thread is only responsible for display; network callbacks use Platform.runLater to return to the JavaFX thread
    for updating controls.

VIII. AppContext (dependency entry point)
  - Call AppContext.install(AppContext.createDefault()) at the beginning of MonopolyDealApplication.start.
  - Each ScreenController obtains the shared GameEngine and SaveGameService through AppContext.get(),
    avoiding scattered manual singletons.

================================================================================
