Root package: com.example.monopoly_deal_game.view
================================================================================
This file describes "what each class / interface / enum in the view package should be responsible for";
specific APIs are subject to the source code.

--------------------------------------------------------------------------------
I. Root package com.example.monopoly_deal_game.view
--------------------------------------------------------------------------------

GameObserver (interface)
  - Function: Observer callback. When the game state (GameSession) in the logic layer/engine changes,
    notify implementing classes to refresh the interface.
  - Should NOT do: Do not write FXML loading here, do not directly modify model collections;
    implementing classes typically call Coordinator or update controls read-only.

GameplayUiBundle (record)
  - Function: Group the pre-divided container blocks in the game FXML (gameRoot, hand area, card piles,
    discard, actionLayer, menu overlay, etc.) into a set of references.
  - Should NOT do: Do not draw specific cards, do not play animations;
    only "holds Pane references" for other places to attach child Nodes.

GameplayViewCoordinator (class)
  - Function: View coordination for a game screen. Holds GameplayUiBundle; provides motionContext() for animations;
    provides sceneResolver() for layer resolution;
    refreshFromSession(GameSession) redraws hands/opponents/sidebar etc. based on the session
    (fill in implementation once agreed with logic).
  - Should NOT do: Do not handle button @FXML events (those are in controller); do not write game rules.

package-info.java
  - Function: This package's MVC responsibility and relationship with controller/model/logic (Javadoc module page).

PACKAGE.txt (other than package-info)
  - Function: Human-readable package description in the same directory (complements package-info).


--------------------------------------------------------------------------------
II. Subpackage com.example.monopoly_deal_game.view.cards (what cards "look like")
--------------------------------------------------------------------------------

AbstractCardNode (class, extends Region)
  - Function: Root node for a single card in the UI. Unified style class name, minimum size;
    subclasses implement renderFrom(Card) to draw the card face based on the model;
    showBack(boolean) for card back/face toggle.
  - Should NOT do: Do not write shuffle/draw path animations (those are in view.animation); do not call GameEngine.

PropertyCardNode (class)
  - Function: Visual for property cards (including dual-color, Wild): color bar, rent tier, full set indicator, etc.,
    data from PropertyCard fields.
  - Model correspondence: com.example.monopoly_deal_game.model.cards.PropertyCard

ActionCardNode (class)
  - Function: Visual for action cards: display title and description thumbnail by action type (Rent, Sly Deal, etc.);
    data from ActionCard and subsequent sub-fields/enums.

BankCardNode (class)
  - Function: Visual for money cards: denomination, color scheme, layout; data from BankCard.

CardNodeFactory (class, static factory only)
  - Function: Accept model.Card as input, create the corresponding *CardNode by type and call renderFrom,
    centralizing instanceof/switch to avoid branching throughout the controller.

package-info.java
  - Function: Description of this package's parallel relationship with the model Card family and layering with animation.


--------------------------------------------------------------------------------
III. Subpackage com.example.monopoly_deal_game.view.animation ("how things move")
--------------------------------------------------------------------------------

UiMotion (interface)
  - Function: Abstraction for a playable UI animation segment. play(MotionContext, onFinished) executes the callback
    after the animation ends normally (e.g., refresh hand card list).

MotionContext (record)
  - Function: Pane references (card piles, hand, discard, overlay, etc.) + default duration needed by animations,
    avoiding excessively long constructor parameters for each animation class.

AbstractUiMotion (class, implements UiMotion)
  - Function: Template method. Subclasses implement buildAnimation(MotionContext);
    the base class play wraps the completion callback and plays.

ShuffleMotion (class)
  - Function: Shuffle animation (visual only; actual card order is determined by logic).
    buildAnimation performs shake/rearrange on child nodes of the card pile area.

DrawMotion (class)
  - Function: Animation of "drawing a card" flying from the card pile area to the hand area;
    may include reparenting, path, or translation.

PlaceMotion (class)
  - Function: Place/land animation for a card landing in a target area (bank bar, property area, discard pile, etc.);
    the target Pane can later be extended into MotionContext or a caller closure.

MoveMotion (class)
  - Function: Translation/reparenting of already-existing cards on the tabletop (e.g., stealing a card, moving when paying),
    distinct from DrawMotion which draws a new card from a pile.

package-info.java
  - Function: Description of how animation is decoupled from CardNode and composed with Transition.


--------------------------------------------------------------------------------
IV. Subpackage com.example.monopoly_deal_game.view.scene ("which layer" in-game)
--------------------------------------------------------------------------------

UiSceneLayer (enum)
  - Function: Logical layer enum within the game interface: main table TABLE, target selection / card selection overlay
    ACTION_MODAL, pause menu MENU_OVERLAY.
  - Note: This is not a replacement for the entire JavaFX Application "Scene";
    it is only a layer concept within the same screen.

ScenePaneResolver (class)
  - Function: Map UiSceneLayer to a specific Pane in GameplayUiBundle,
    making it easy to attach child nodes for popups/full-screen animations onto the correct container.

package-info.java
  - Function: Description of the mapping between in-game layers and FXML regions.


================================================================================
Boundary with controller (avoid reinventing the wheel)
================================================================================
  - User clicks, overall scene switching, AppContext engine retrieval: in the controller package.
  - Card face Nodes, trajectory animations, refreshing Pane contents per session: in the view package
    (driven by GameplayViewCoordinator or GameObserver implementing classes).
